O que é este projeto?

O Rick & Morty Explorer é uma aplicação full-stack desenvolvida do zero como projeto de portfólio, com o objetivo de demonstrar conhecimentos em desenvolvimento backend com Java e Spring Boot, desenvolvimento frontend com Angular, integração entre sistemas via API REST, boas práticas de código, testes unitários e deploy em produção.

A aplicação consome a Rick and Morty API - uma API pública e gratuita que disponibiliza dados sobre personagens, episódios e locais da série animada Rick and Morty. Em vez de consumir diretamente no frontend, foi criada uma camada de backend própria em Java que atua como intermediária, adicionando cache, validações e documentação.

Como o backend funciona?

O backend foi desenvolvido com Java 21 e Spring Boot 4.0.3, seguindo a arquitetura em camadas: Controller → Service → Client. O RestClient - client HTTP moderno introduzido no Spring 6.1, é responsável por consumir a API externa. Cada chamada passa pelo Spring Cache com Caffeine, que armazena os resultados em memória por 10 minutos, evitando chamadas repetidas e melhorando a performance.

O GlobalExceptionHandler intercepta todos os erros e retorna respostas JSON padronizadas com status HTTP correto, em vez de stack traces. A documentação dos endpoints é gerada automaticamente pelo Swagger/OpenAPI e pode ser acessada via interface interativa. Os testes unitários com JUnit 5 e Mockito cobrem os principais cenários do service, incluindo validações de entrada e comportamento esperado.

Como o frontend funciona?

O frontend foi desenvolvido com Angular 19 e Angular Material, com tema escuro espacial inspirado no universo da série. Os services encapsulam todas as chamadas HTTP ao backend usando o HttpClient do Angular, retornando Observables que os componentes assinam via subscribe. O FormsModule com ngModel permite a busca em tempo real com debounce de 500ms, a chamada à API só é disparada quando o usuário para de digitar, evitando requisições desnecessárias.

O painel de detalhes do personagem aparece dinamicamente dentro do grid, logo abaixo da linha do card clicado, calculando a posição com base no índice do item e no número de colunas da tela. O sistema é responsivo e ajusta o número de colunas automaticamente conforme o tamanho da tela, garantindo que o painel sempre apareça no lugar correto em qualquer dispositivo.

Como está hospedado?

O backend roda em um container Docker na Oracle Cloud Free Tier, em uma VM Ubuntu 22.04 com 1 OCPU e 1GB de RAM. O Nginx atua como proxy reverso, redirecionando as requisições da porta 443 (HTTPS) para a porta 8080 do container. O certificado SSL é gratuito, gerado pelo Let's Encrypt via Certbot, com renovação automática configurada. O domínio é gerenciado pelo DuckDNS, um serviço gratuito de DNS dinâmico.

O frontend está hospedado no Vercel, que detecta automaticamente os commits no GitHub e faz o redeploy sem nenhuma configuração adicional. A comunicação entre frontend e backend é liberada via CORS configurado no Spring Boot, permitindo apenas as origens autorizadas.

Principais desafios

O maior desafio foi o deploy do backend, onde foram testadas quatro plataformas diferentes antes de chegar à solução final. O Railway tinha trial limitado, o Render dormia após inatividade, o Fly.io expirava em 7 dias. A Oracle Cloud resolveu todos esses problemas, mas exigiu configurar manualmente a VM, o Docker, o Nginx e o SSL, um processo que ensinou muito sobre infraestrutura em nuvem e DevOps.

Outro desafio foi o bloqueio de Mixed Content, o frontend em HTTPS não conseguia chamar o backend em HTTP. Isso só foi resolvido após configurar o domínio com HTTPS via Let's Encrypt. O Zone.js também causou um erro silencioso no Angular 19, que não o inclui mais por padrão e precisa ser configurado manualmente. O Lombok teve incompatibilidade com o Spring Boot 4.0.3, exigindo que todos os getters, setters e construtores fossem escritos manualmente.

⚠️ Dificuldades encontradas

01
Incompatibilidade do Lombok com Spring Boot 4.0.3
O Lombok não funcionou corretamente com o Spring Boot 4.0.3 no IntelliJ, impedindo a geração automática de getters, setters e construtores. A solução foi escrever todos os métodos manualmente, o que na prática ajudou a entender melhor o que o Lombok faz por baixo dos panos.
02
Deploy do backend - múltiplas tentativas
O Railway foi a primeira opção mas o trial expirou. Em seguida passei a usar o Render, mas ele tinha problema de sleep após inatividade. Vi sobre o Fly.io, mas ele durava só 7 dias de teste gratuito. A solução final foi a Oracle Cloud Free Tier com Docker, Nginx e certificado SSL via Let's Encrypt, solução mais robusta e um mês de teste gratuito.
03
Mixed Content - HTTP vs HTTPS
O frontend no Vercel usa HTTPS mas o backend estava em HTTP puro. Navegadores modernos bloqueiam chamadas HTTP dentro de páginas HTTPS. A solução foi configurar o Nginx com proxy reverso e certificado SSL gratuito via Let's Encrypt, deixando o backend acessível via HTTPS.
04
CORS entre frontend e backend
Configurar o CORS corretamente para permitir que o Angular em produção no Vercel se comunique com o backend foi um desafio, especialmente após cada troca de plataforma de hospedagem que gerava uma nova URL que precisava ser adicionada ao CorsConfig.
05
Zone.js no Angular 19
O Angular 19 não inclui o Zone.js por padrão — ele precisa ser instalado e configurado manualmente no angular.json via polyfills. Isso causou um erro de inicialização que demorou para ser identificado.
